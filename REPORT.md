Итоговый отчёт по мини-проекту "Угадай число"

Студент: Матисов Артем Алексеевич
Тема: игра, в которой компьютер загадывает число, пользователь угадывает, ведётся список лучших попыток.


Путь по этапам

Этап 1 — Инициация и требования. Определена тема и составлены пять пользовательских историй, покрывающих игровой процесс, выбор сложности, подсказки, таблицу рекордов и личный результат. Сформирован MVP: одна партия с сохранением результата и просмотром таблицы. Задачи оформлены в GitHub Issues и распределены по Kanban-доске с колонками Backlog, In Progress, Review, Done.

Этап 2 — Проектирование. Приложение разделено на три модуля: game.py содержит чистую логику без зависимости от ввода-вывода, db.py реализует хранение через единственную таблицу scores в SQLite, main.py соединяет их консольным меню. Схема взаимодействия: пользователь — меню — логика раунда — сохранение результата — файл scores.db. Архитектура зафиксирована в README до начала кодирования.

Этап 3 — Разработка по TDD. Разработка велась строго по циклу red-green-refactor. Первыми написаны тесты для generate_secret_number и check_guess, затем их реализация. Следом — validate_guess, get_hint, play_round. Параллельно разрабатывался слой db.py с тестами через фикстуру tmp_path, исключающую пересечение тестовых баз данных. Каждая функция оформлялась отдельным Pull Request.

Этап 4 — Приёмочное тестирование и стабилизация. Пройден чек-лист из 15 пунктов вручную. Обнаружены два дефекта: пустое имя игрока вызывало необработанное исключение (BUG-01), а функция validate_guess не обрабатывала тип None (BUG-02). Оба дефекта исправлены после написания воспроизводящих тестов. Приложение доведено до версии 1.0.0.

Этап 5 — Сопровождение. Обработано пять обращений: два бага (исправлены в 1.0.1 и 1.0.2), одно улучшение (реализовано в 1.0.3), один запрос на изменение (обоснованно отклонён), один вопрос по документации (закрыт обновлением README).

Этап 6 — Ретроспектива и отчёт. Текущий документ.

Ответы на вопросы ретроспективы

Что было самым сложным в тестировании?

Сложнее всего оказалось тестировать функцию play_round, которая по природе своей интерактивна: она читает ввод через input и пишет вывод через print. Прямой вызов такой функции в тесте потребовал бы патчинга встроенных функций через unittest.mock, что делает тесты хрупкими и многословными. Решение — инъекция зависимостей: функция принимает input_func и print_func как параметры со значениями по умолчанию input и print. В тестах они подменяются на lambda и list.append соответственно. Это позволило полностью протестировать логику раунда без единого патча.

Как изменилось бы приложение, если бы все баги были известны заранее?

Если бы с самого начала была известна проблема с пустым именем, проверка добавилась бы в run_game() на этапе 3, а не как фикс в 1.0.1. Аналогично для None в validate_guess — кортеж (ValueError, TypeError) был бы написан сразу. Это сэкономило бы два цикла патч-выпуска. В целом заблаговременное знание о краевых случаях не изменило бы архитектуру, но убрало бы корректирующие коммиты и сделало историю репозитория более прямолинейной.

Чему научил процесс поддержки?

Поддержка показала несколько вещей. Пользователи находят краевые случаи, которые кажутся разработчику невозможными (кто же введёт None?). Отказ от реализации запроса CHG-01 оказался не менее важным решением, чем реализация ENH-01: не каждое пожелание улучшает продукт. Вопрос QUE-01 выявил пробел в документации — функциональность была, но не описана. Это напомнило, что полезность фичи определяется не только её наличием, но и тем, знает ли о ней пользователь.


Ретроспектива

Что удалось хорошо:
строгое следование TDD дало 100% покрытие логики до этапа приёмочного тестирования;
инъекция зависимостей в play_round упростила тесты и сделала код гибче;
чёткое разделение на три модуля позволило тестировать каждый слой независимо;
все обращения поддержки обработаны с написанием теста до исправления.

Что можно улучшить в следующий раз:
добавить type hints ко всем функциям с самого начала;
написать интеграционные тесты для main.py, имитируя полный сценарий партии;
добавить логирование вместо print для упрощения диагностики;
рассмотреть использование dataclass для структуры записи результата вместо tuple.
